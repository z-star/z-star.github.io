jvm 调优

2015-05-11
javajvm

1.程序计算器:CPU切换时候保持了线程运行的环境上下文。线程私有.这也是JAVA虚拟机里唯一没有OOM的
2.Java虚拟机栈(Stack Frame):线程独有的局部变量表，操作数栈。生命周期和线程一样。也会OOM
3.本地方法栈(Native Method Stack):跟虚拟栈一样，只不过对应的是Native方法服务。
4.堆(java heap):jvm共享的内存区域，对分配对象实例的区域，垃圾回收的主要管理区域。所以从GC角度来看，还会分为新生代：Eden空间，From Survivor,To Survivor等，这个区域也会OOM。
5.方法区(Method Area):用于存储已被虚拟机机加载的类信息，常量，静态变量。是heap的一个逻辑部分。1.7之前方法区保持常量池所以需要限制大小MaxPermSize,后来这个参数可以取消了。
6.运行时常量池(RunTime Constant Pool):用处存储符号应用，String就存在这里。
7.直接内存(Direct Memory):主机的内存不属于JVM的内存,NIO中可以使用Native函数分配堆外内存。DirectByteBuffer。所以这个直接内存其实也会OOM

OOM总结
1.java heap space:
大部分在heap正常大小的情况下是重复创建对象了。可能有循环创建对象的可能。内存泄漏，打开dump出来的错误文件快照。对象的GC Roots引用链信息定位泄漏代码。

2.StackOverflowError:虚拟机栈中，栈帧大于容量就会发生这样的情况，查看线程方法的代码。

3.动态加载类glib之类过多的情况下PermGen(永久带)也可能发生溢出的情况。常量池溢出

4.直接内存溢出:MaxDirectMemorySize,不指定就跟-Xmx 大小一样，一边程序dump出来的文件很小，冰山对象，只是DirctByteBuffer 这个壳而已。通信数据最终都会被copy到堆外。所以netty 等使用的buffer 都是堆外内存，因为只使用一次。不断循环利用这块内存。没有指明区域，而且dump 出来的文件没有异常，就可以查看NIO问题了。

5.native memery 需要做减法排查。

GC
1.判断对象是否可回收：
GC Roots,与GC ROOTS,没有连接的对象。可以被回收。GC Roots的对象包含下面几种：
1）虚拟机栈中的引用对象。
2）方法区中类静态属性引用对象
3）方法区中常量引用的对象
4）本地方法栈中JNI
应用对象强弱自分方便回收等级
1）强引用(Strong Reference):Obejct obj = new Object() 这类的引用，只要强引用还存在，就不会被GC
2) 软引用(soft Reference):SoftReference来实现软引用，在内存溢出之前回gc
3) 弱引用(weak reference):weakReference来实现弱引用，比软应用更低，下一次垃圾回收就会被回收掉
4) 虚引用(phantom Reference):PhantomReference来实现虚引用，最弱的引用，对一个对象设置虚引用是为了得到它被GC的系统通知。

对象回收要经过两次标记。
回收方法区和常量池，常量池中的常量没有引用的情况下可以回收，这个简单，但方法区中对一个类的回收就比较复杂。在大量使用反射，动态代理，CGLIb等这类平凡自定义classloader的场景需要聚类类的卸载功能，保证方法区不OOM。参数-verbose:class 和-XX：+traceClassLoading

GC算法
1.标记-清除算法(mark-Sweep)标记清楚算法。
最早的算法，后续算法都是对他的改进，主要就是标记处需要GC的区域，第二部就是清楚掉这些区域。步骤简单，主要有两个不足之处:一个是效率问题，标记清除两个过程效率都不高，另一个是空间问题，后续会产生大量的不连续碎片。空间碎片太多会导致大对象没有连续的空间分配而提前触发GC。
2.复制算法(copying)
复制算法就是讲内存分为两块，一块使用，一块用来放置没有被回收的对象。这样就不会出现不连续问题，但内存使用就只有一半。现在这种算法主要采用在新生代上。而比例不是1：1而是划分成eden空间和两块较小的Survivor空间。这样Eden和Survivor中存货下来的对象就会复制到剩下的一个survivor空间中。按照8：1的比例，空间就浪费了10%。但这个survivor区域放不下了，就需要放到老年代去。
3.标记-整理算法(Mark-compact)
标记整理算法第一步也是标记，但第二步是先让存活的对象往一端移动，让后直接清理掉端边界以外的内存，这样就解决了内存不连续的问题了。
4.分代收集
年轻代合适复制算法，因为大量对象中活下来的不多。所以只需要少量复制对象的成本就可以。老年存活率比较高。所以才有标记的两个算法。

垃圾收集器
young Generation:Serial,ParNew,Parellel Scavenge,G1
old Generation:CMS,Serial Old(Msc),Parallel old,G1

1.Serial 最大的特点就是”Stop the World”,单线程收集器，新生代是复制算法，老年代是标记整理算法。都会采用暂停所有用户线程来进行回收。Serial用在客户端模式的虚拟机下是一个不错的选择，对响应比较敏感的就算了。
2.ParNew 是Serial的多线程版本，使用多个线程进行回收，参数都有很多是共用的(-XX:SurvivorRatio,-XX:PretenureSizeThreshold,-XX:HandlePromotionFailur),他也会暂停所有用户线程来做GC，除了serial以外只有他能和CMS一起工作，所以为什么它一直存在的价值。-XX:ParallelGCThreads来限制垃圾收集的线程数量。
3.Parellel Scavenge:目的不尽可能的减少停顿时间，而是提高吞吐量，也就是降低垃圾回收时间的比例。-XX：MaxGCPauseMillis -XX：GCTimeRatio来控制吞吐量的。所以他也叫做吞吐优先收集器。还可以设置-XX:+UseAdaptiveSizePolicy,这样虚拟机会收集运行情况动态调节，新生代大小，Eden，survivor区域，等，所以这是一种自适应的调节策略。自适应调节策略也是他与ParNew的对大区别
4.Parellel Old:是Parellel的老年版，配合起来比较适合重视吞吐量和自适应的场景。
5.CMS:以停顿时间最短为目的的回收器，主要和ParNew配合使用，很多Server端都用这样的回收组合。使用的是标记清除算法，前面的老年代的基本都是标记整理算法。但他使用的标记清除比较复杂。3个标记1个清除。
1）初始标记(CMS initial mark)
2) 并发标记(CMS concurrent mark) 并发标记牺牲了，空间，也照成浮空垃圾。会造成CMS不可用需要替代算法。
3) 重新标记(CMS remark)
4) 并发清除(CMS sweep)
三个缺点，
1.因为是有并发操作，所以线程数是 (cup+3)/4，但当CPU小于4个时候，基本就变成非并发了，所以需要在多核的情况下才能体现优点
2.-XX:CMSInitiatingOccupancyFraction=75启动gc阈值设置。1.6默认是92%，如果gc期间预留的内存无法满足需需要就会报错“concurrent Mode Failure”;
3.cms是标记清理，所以会有参数指定整理的过程-XX:+UseCMSCompactAtFullCollection这是默认开启的，还有-XX:+CMSFullGCsBeforeCompaction(默认是0，表示每次进入Full GC时都进行的碎片整理)
6.G1：内存物理划分不再是新生代，老年代。而是划分成多个大小相等的独立区域(region),G1后台维护了一个优先列表，每次都回收价值最大的region.保证在有限的时间内可以获得尽可能的高效。

GC日志解读
33.123：[GC[DefNew:3324K->152K(3712K),0.0025925 secs] 3324K->152K(11904K),0.0031680]

最前面的“33.123”虚拟机启动以来的时间,后面的“DefNew”表示发生的区域，这里是Serial的新生代收集器”Default New Generation”,如果是ParNew就变为”ParNew”,Parellel Scavenge就是”PSYoungGen“,冒号右面的数字”3324K->152K(3712K) 表示新生代的GC情况 3324K为GC前的内存使用箭头后面为GC后的内存使用括号内为该区域内存总量。后面的”3324K->152K(11904K)”就是整个heap中的情况。所以新生代回收的量等于整个gc回收的，才说明这次GC都是新生代，要不然就说明有进度old和perm的。

排序算法总结

2015-05-11
java

冒泡排序 时间复杂度O(n2) 空间复杂度 O(1)
比较相邻的两个数，前一个数大于后一个数就交换，知道比较到最后一个，最后一个将是最大的数。下一次继续比较但排除最后一个数。知道全部排完。
冒泡的关键是最大，或者最小的数，最后都会出现在最后面。依次减少一位，所以时间复杂度就是O(n2)，稳定的。

选择排序 时间复杂度O(n2) 额外空间复杂度O(1)
固定一个位置，来做定位，顺序选择后续所以的位跟他比较，如果大于这个位置的数，就交换。一轮选完，后这个位数肯定是最大的，下一轮排除。

插入排序O(n2) ** 从第二位开始整理扑克牌。所以for(int i=1;i<size;i++) 开始比较。
//把i前面所有大于要插入的牌的牌往后移一位，空出一位给新的牌
1.从第一个元素开始，该元素可以认为已经被排序
2.取出下一个元素，在已经排序的元素序列中从后向前扫描
3.如果该元素（已排序）大于新元素，将该元素移到下一位置
4.重复步骤3，直到找到已排序的元素小于或者等于新元素的位置
5.将新元素插入到该位置中
6.重复步骤2
每一次都会移动数据。

希尔排序是插入排序的一种高速而稳定的改进版本。

public void sortHeap(int[] sort){
	for(int i =sort.length-1;i>0;i--){
       int temp = sort[0];
       sort[0]=sort[i];
       sort[i]=temp;
       maxHeap(sort,i,0);
    }
}
public void buildMaxHeap(int[] sort){
   int start=getParent(sort.length-1);
   for(int i =start;i>=0;i--){ //从最后一个父亲节点开始往前，遍历所有的树，构成最大堆。
    maxHeap(sort,sort.length,i);
   }
}
private void maxHeap(int[] sort,int heapSize, int index){
    int left = getLeftChild(index);
	int right = getRightChile(index);
	int largest = index;
	if(left<heapSize&&sort[largest]<sort[left]){
        largest =left;
	}
	if(right<heapSize&&sort[largest]<sort[right]){
        largest =right;
	}
	if(largest!=index){
	  int temp = sort[index];
	  sort[index]=sort[largest];
	  sort[largest]=temp;
	  maxHeap(sort,heapSize,largest); //子树可能不是最大堆了。继续往下。
	}
}
private int getParent(int child){
	return (child-1)>>1;
}
private int getRightChild(int parent){
	return (parent<<1)+2;
}
private int getLeftChild(int parent){
	return (parent<<1)+1;
}

分布式系统介绍

2015-03-24
分布式系统

1.在分布式环境中，先从多线程讲起。

多线程基本分三种模式：互不通信多线程模式，共享容器协调的多线程模式，事件协调的多线程模式。

2.网络IO

ISO(5层) 物理层->数据链路层->网络层->传输层->应用层 (会话层->表示层 是7层协议)


一次io操作过程分为两个阶段数据等待，也就是执行任务阶段。另一个就是数据拷贝阶段。两个阶段。
BIO:blocking io 最直白的io，所有io阶段都要阻塞等待返回。
NIO: None-blocking io 非阻塞io,在数据等待阶段轮询检查,而不是阻塞。 new io.采用io复用，Reactor模式，由一个系统线程来做数据等待过程，用户线程不会被阻塞，当任务处理结束，来到数据拷贝阶段，再通知客户端，拷贝数据，这个时候是阻塞的。
AIO: 完全异步的io,从数据等待，到数据拷贝都由系统完成。

3.负载均衡
硬件负载均衡：硬件作为负载均衡的中间件，来分发请求。
lvs:软件来作为中间件，分发请求。
名称服务：做个名称服务器来作为服务器地址提供。让客户端和服务端直连。
规则服务：规则服务会发送规则给客户端，让客户端更具规则跟服务端直连。跟名字服务器一样，只不过交流方向不一样。
master:master服务器不是把规则传给客户端，而是规则计算好了，直接告诉客户端访问那个服务端。

4.分布式系统的要点。
1）缺乏全局时钟，也就是控制时序难。故障独立性。不要有单点功能，或者说是单点故障。事务的挑战.

2）分布式下的session问题解决方案：
session sticky方式：这种方式需要负载均衡器根据每次请求的session来转发对应服务器。缺点很明显，就是让负载均衡器来解析很消耗。让负载均衡变成有状态的。这样内耗很大，并且如果服务器重启了，那session数据也就没有了。
session replication：集群之间相互备份session数据，这样的方式集群一大了就完蛋了，只适合非常小的集群。
seesion集中存储：这种方式在大型网站比较常见。缺点也就是有网络开销和，多余的请求。还要保证数据的稳定性。
Cookie Based：在cookie里写session。这种方式比较浪费http的请求消耗，而且不安全，另外session会被cookie所限制。但这样的方式是最简单暴力的，在少部分地方可以使用。
3）演变过程
数据库读写分离 问题：数据复制问题。数据复制延迟。应用对于数据源选择问题。
搜索引擎添加:搜索请求不与mysql等关系型数据库来承担。（es solr 等）
数据缓存：(redis memecache等)加大读能力。问题：实时更新。吃内存。数据变更时失效，和数据变更时更新。
页面缓存：ESI模块
分布式存储系统：弥补关系型数据库的不足。hdfs hbase reids.
数据库垂直，水平切分：垂直切分，也就是更具不同的业务切分不同的库，水平切分也就是同一个业务切分成多个数据组。数据库问题解决后面就是应用问题了。
应用拆分：服务化之路 基本三层 web服务系统，业务中心，底层数据。这里就开始需要消息中间件了。
4)动态代理和反射，动态代理，让客户端像本地服务一样调用远程服务。
中间件：服务框架：dubbo。分布式数据层Druid.消息中间件：qmq,配置管理中心：软负载中心：zk,数据复制迁移。
服务化框架要点：
1.服务框架与应用还有容器之前的关系。dubbo是第三方jar.更应用一起部署。totem是容器。本省就有容器的功能。还有一种是引入到容器类中。
2.接口，方法，参数的路由。接口路由有利于让慢响应接口不影响全局。方法路由也是。多机房动态分配路由规则。还有一个就是流控。
3.序列化基本使用json.协议的扩展性是比较重要考虑的。标明版本以便未来扩展。客户端网络选择一般使用Nio中的future方式。
4.服务端如何暴露服务，本地服务的注册管理和请求定位服务并执行。
5.协议解析后得到序列化数据后，反序列化在什么线程中执行，反序列化后的定位服务后，调用服务在上面线程执行者都是需要考虑的。线程隔离也是通过接口，方法，参数等，来确定不同的线程池。实现线程隔离。服务端也可以进行限流和分类。限流用RateLimter，用令牌桶来实现。一段时间间隔发一批令牌，用完就得等待，或者没有就返回false.
6.服务升级，非接口更改，与添加接口方法都很简单，不用管调用者。接口参数变更会比较麻烦。一起修改服务不太现实，版本号能解决这个问题，参数可扩展会让参数变成类似m的结构会让校验变得比较复杂。并且不直观。
7.请求合并，分布式环境中请求合并，加分布式锁，会比较复杂和开销大，可以通过路由的方式把想要的请求合并到单机上去。
8.服务治理：读和写，管理和查看。 服务信息：服务编码，编码注册，编码定位服务信息。服务质量：最好，最差的服务排行，服务质量趋势,各种查询条件的支持。服务容量：服务容量和当前水位的展示，历史趋势图，根据水位的高低排序。服务依赖：依赖服务展示，被依赖服务的展示，依赖变化。服务分布：服务在不同机房的分布，分布服务列表。服务统计：调用次数统计排名趋势，出错统计排名趋势，响应时间统计排名趋势。服务元数据：服务的方法和参数。服务查询：服务索引的应用，发布时间，地址列表，容量，质量，调用次数。服务报表,服务监控。
9.服务上下线：服务所有机器的上下线，针对机器的上下线，DoubleCheck控制。服务路由：路由管理界面，dc控制。服务限流降级。服务归组。服务线程池管理，服务授权。
分布式数据层要点：
1.垂直拆分遇到的问题。单机的ACID（原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability））被打破。可能需要分布式事务，join和外键操作受限制。水平拆分遇到问题，同样是ACID被打破，join,外键，单库的自增id,单个逻辑意义上的查询需要夸库了。
2.AP,RM,TM 之间的关系，当需要TM管理的时候，AP和RM之间就使用XA-naitve api交互。而不是普通的native-api交互。AP和TM之间有TX接口对，启动，提交，回滚操作的控制。TM通过AX来和RM 交互。管理RM的链接。
3.两阶段提交2pc(two phase commitment protocol)之所以称为两阶段提交，单库直接提交回滚，而在分布式环境中需要多一个准备阶段，所有单节点都准备好了，再一起提交。或者有一个节点没准备好，就回滚。
4.CAP(consistency availability partition-Tolerance) 一致性，无论在什么样的情况下所有节点的数据都是一致的。可用性，就是无论什么情况下系统一定要有响应。分区容忍性，系统一部分出现问题，系统任然能工作。
CA，一致性，可用性，这其实就是传统的单机。AP，放弃一致性，最求容忍性和可用性，这就是很多NOSQL架构的选择。CP，放弃可用性。网络问题可能让整个系统不可以。
5.BASE basically Available 基本可用，允许分区失败。soft state 软状态，一段时间状态不同步。Eventually consistent最终一致性。
6.比2pc轻量一些的paxos,paxos模拟的希腊城邦，模拟议会决议的方式。完成一致性。
proposers:提出议案的角色，acceptors:收议案进行判断的角色。多数人接受议案则该议案通过。通过的议案只能学习，learners。proposal 议案，value，决议,每一个议案都会有{编号，决议}对组成。paxos算法实例中，一次只能批准(chosen)一个决议(value),learners只能获得被批准(chosen)的value.
议员记录的信息：lasttried[p]议员p发起的最后一个议案的编号，没有任何发起则为负无穷大。PreviousVote[p]议员p投票的所有表决中，编号最大的表决对应的票数。NextBallot[p]由议员P发起的所有LastVote(b,v)消息中，表决标号的最大值。
7.集群内数据一致性算法。quorum 和vector clock quorum 一致性和可用性三个变量：N：数据副本数，R 成功读的最小节点数。w:成功写的最小节点数。
8.唯一自生产ID 类似uuid ，但问题是不连续而且太长，ID集中管理：请求消耗性能，生成器稳定性，存储问题。在订单库中使用的是偏移量来hash获取订单的库和响应的表名。
消息中间件：
1.Queue模型，点对点的方式，（PTP PEER TO PEER）发送者将消息发送到队列queue中，任意消费者消费后消息从队列中去除，所以只有一个消费者能消费到。
2.Topic模型, 也叫订阅模式 pub/sub 同过个group的消费者，可以收到所有topic订阅的内容。
3.一致性：消息状态存储，本地存储和broker存储，本地存储的侵入性会更强，因为事务是在本地完成的。但消息会高，broker存储消息状态，需要由本地发送消息并等待broker回复。假如消息投递失败了，消息状态还在远程broker。这样大大增加了broker的使用量。
本地数据库配合保证一致性。业务操作之后，消息落地到本地库，在同一个事务中。事务提交之后，开始发送消息，消息的发送回调成功删除本地消息。消息发送失败，会重试发送3次，因为这可能是网络问题。或者阻塞，或者超时。活更新本地消息状态。等待task 再来处理。
非本地的情况下，发送消息状态到broker,broker落地消息，返回消息记录成功给发送者，发送者做业务操作，做完发送消息给broker,broker更改消息状态。
这种情况下还需要一部验证，也就是长时间没有收到消息更改状态，需要broker去，询问发送者消息的状态。这样的情况下 发送成功与否的检查都在中间件也就是broker这边。业务和中间件是强耦合，并且直接影响业务，消息中间件的可靠性要求非常高。可靠性需要平衡，需要让业务端也承担更多的可靠性。他要做很多跟业务之间的检查工作，这样的方式不靠谱。
本地消息落地的数据表设计

当消息已经确认发送到broker之后，还要经过一次消息落体，来保证消息一定发送给消费者，这里是一对多的关系，也就是有多个消费者。消费者对消息状态的修改必须是隔离开的。把消息的所有状态全部隔离在redis当中，同一条消息，不用的group，有不同的消息状态。
broker的库的扩展比较容易，因为消费者没有一个主动获取消息查找的功能，也就是不需要某个消息到底在那个库的功能。所以可以任意扩展。这是无状态的。
消息发送的一步流程：
生产者在写业务数据的事务中一起写消息，把消息落地(等待后续删除)，事务提交以后发送消息。消息放入队列，异步处理，等待分组发送(不同的subject隔离)到不同的broker。消息落地之后向producer返回结果，让producer删除消息，理想情况下只有一份消息落地。根据不同的订阅grop 发送消息给消费者。
消息broker发送到consumer，consumer接收到之后返回，异步task处理业务。处理完之后ack.
保证消息的局部顺序：
push 变成pull  就能让消费者自己来控制消息的顺序。这个非常合理，有时候思想固化了，得跳出原来的设定。一定要回头看看。
push  broker状态保持了数据传输的状态，重试等维护也是borker来做。有点就是实时性比较高。流控也要造broker来做。
pull  状态都由消费端来控制，实时性完全由轮询时间来决定。不用做流控，自己能消费多少，消费多少。

软件负载中心：
1.聚合数据，提供服务者的信息，也就是服务提供者的聚合信息，dataId:group:value dataId相当于一个提供的服务名，subject,group，相当于有多多少个单个的分组，value 就是具体的地址信息。
2.订阅关系：提供的是dataID:group：conusmerGroupId, 是一个订阅者和suject的订阅关系，作用是能及时提醒所有的订阅者，提供者信息的变更。
3.链接数据：groupId:connection 每一个group的维护，发生变化来通知。
4.根据上面三个数据，围绕的就是服务提供者链接地址的暴露，订阅者关系的暴露，服务者链接维护，当出现情况，及时通知订阅者。
配置中心：
1.客户端只能读取数据，并且有本地文件做容灾，因为是非状态数据。链接方式是http长轮询，也就是链接后，有数据返回，无数据等待数据，知道超时，继续新链接。
2.服务端，会用nginx 来缓存文件。应用自身还拉倒本地，并且再缓存一次，这就是client端要做的。版本快照。文本格式。
