分布式链路跟踪调度系统

2015-07-12
tools

面对日益复杂的大规模分布式集群环境，入口流量涨了，不知道要升级扩容哪些后端服务；数据库撑不住了，不知道是哪些应用读写多了；订单不见了，付款成功出不了票了，grep n多台机器日志，联系n个运营团队也没找到答案；这些问题都可以交给分布式链路跟踪调度系统来解决。

分布式链路跟踪调度系统源于Google Dapper论文，继Twitter的ZipKin，eBay的Centralized Activity Logging (CAL)，淘宝的鹰眼，京东商城的Hydra之后，它的强大之处在于能做到追踪每个请求的完整调用链路，收集调用链路上每个服务的性能数据，并与异常监控集成，在收到报警时能从监控峰值点关联上若干异常trace样本，快速定位问题。相信在不久的将来，它能通过计算和比对性能数据，分析系统瓶颈点，并最终反馈到服务治理中。

想知道它是怎样在你的业务流程中悄无声息的埋点监控的吗？
想知道怎样使用它就能快速定位问题，轻松搞定线上环境各种疑难杂症吗？

想知道哪些业务线已经接入并从中收益吗？

应用中心

2015-07-12
tools管理平台

应用中心
  • 维护应用/组织/人之间的关系
  • 基础系统交互 应用启动时，通知其他系统
  • 应用资源访问控制 如：维护应用与应用之间接口访问的HTTP和Dubbo接口白名单

App应用验证过程
发送app到应用中心进行验证；
验证通过后，返回识别出环境(dev/beta)、主机名等信息，并下发临时服务器令牌(server_token)
对于应用程序，可通过如下方法完成上述步骤。

Web应用
在完成上述App验证过程后，注册服务器端口等信息
根据中心返回的环境类型，选定对应的(dev/beta/prod)应用中心；
发送web应用的服务端口号
应用中心接收请求后，（dev/beta环境从线上同步应用信息），注册主机信息
应用中心广播”应用启动”事件到消息系统
可通过在web.xml中配置ServletWatcher为Filter完成。

  1. 确定哪些内存需要回收
  2. 确定什么时候需要执行GC
  3. 如何执行GC。
学习Java GC机制
  1. 内存是如何分配的
  2. 如何保证内存不被错误回收（即：哪些内存需要回收）
  3. 在什么情况下执行GC以及执行GC的方式
  4. 如何监控和优化GC机制。
