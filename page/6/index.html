set方法就4个操作，第一个
int wordNum = (int)(index >> 6); //计算出坐标所在的long数组的位置。
long bitmask = 1L << (int)index; //计算出移动该坐标后的long的大小
this.bits[wordNum] |= bitmask; //和所在坐标的long进行或运算，也就是并集。

最后总结一下OpenBitSet数据存储的特点：OpenBitSet无法存储重复的数据。数据存储到OpenBitSet中后就是有序的了。OpenBitSet适合存储密集程度高，且量大的数据。OpenBitSet中存储的数据适合位运算，比如取交集、并集、补集

elasticsearch查询类型

2015-10-17
Elasticsearch学习

1.TermQuery:最基础的词条查询。对每一个term的匹配。

2.TermsQuery:多词条查询，相当于多个term的bool查询，默认为or操作。

3.match_all:匹配所有文档

4.commonQuery:常用词查询,把查询词分成两组，一种常用词，另一个非常用的次.只对非常用词，也就是频率较低，比较重要的词进行查询。高频词，也就是不重要的次，只作为第一次低频词查询结果后打分使用。这样实现了高效率。

5.match:布尔值匹配查询,分词的情况下可以使用这个查询,匹配多种情况下的，他也有cutoff_frequency这个选项，也就像常用词查询一样是高低频词的分组情况。

6.match_phrase:布尔短语查询，他与布尔查询最重要的区别在查询词的顺序需要和文本一致的。也就是布尔查询的query 是无序的,phrase是有序的，在slop设置的情况下才允许中间词语有遗漏。

7.match_phrase_prefix:布尔短语前缀查询，跟上一个查询的区别就是query的最后一个词可以只有前缀。感觉使用场景并不多。还可以使用disable_coord来减少查询时间。
8.multi_match:多布尔匹配查询，与match 不用的是可以指定有个字段，来做查询field.

9.query_string:支持lucene DLS查询语法的一个查询.

10.IdsQuery标识符查询:也就是查询id的。

11.PrefixQuery:前缀查询

12.FuzzyQuery:模糊查询,编辑距离查出相近的term 再把这些term进行布尔查询。
13.RangQuery:范围查询
14.dismaxQuery:最大分查询，大分越大。小分越小。
15.RegexpQuery:正则表达式查询
16.constant_score:不计算打分的查询.
17.indicesQuery:索引查询,多个索引之间的条件查询。
18.过滤查询。post查询和非post查询之间的区别。post查询是对每一个query结果的过滤。而非post查询是，先过滤再查询。所以非post查询要更高效。
19.父子查询需要加载标识符，所以有一个预热warm过程可以让父子查询加速。

进程和线程

2015-10-12
java

1.为什么要有进程

进程是为了解决，cpu的使用效率，让系统能够并发操作。这样才能在单台计算机，丰富的体验。因为进程分割内存，并交替使用cpu来运算。才能做到cpu使用率提高。在程序IO的情况下，cpu是闲置的，io时间越长闲置时间越长，多个程序并发操作。切换非常快，人根本没法感知，所以才能让如今的操作系统有很好的体验。

2.为什么要有线程
在出现了进程之后，操作系统的性能得到了大大的提升。虽然进程的出现解决了操作系统的并发问题，但是人们仍然不满足，人们逐渐对实时性有了要求。
换句话说，进程让操作系统的并发性成为可能，而线程让进程的内部并发成为可能。但是要注意，一个进程虽然包括多个线程，但是这些线程是共同享有进程占有的资源和地址空间的。进程是操作系统进行资源分配的基本单位，而线程是操作系统进行调度的基本单位。

3.多线程并发

　　由于多个线程是共同占有所属进程的资源和地址空间的，那么就会存在一个问题：如果多个线程要同时访问某个资源，怎么处理？线程安全问题就这样产生。

2.服务器保持了大量CLOSE_WAIT状态

一般情况下都是代码写错了。
1) httpClient举例子：请求异常，非200得情况下，服务方方发出关闭连接的请求，请求方就是被动关闭连接，也就是CLOSE_WAIT。如果请求方自己没有释放连接，那就造成CLOSE_WAIT堆积。
2) Nginx上面的产生原因还是因为Nagle’s算法加Nginx本身EPOLL的ET触发模式导致。ET出发模式在数据就绪的时候会触发一次回调操作，Nagle’s算法会累积TCP包，如果最后的数据包和FIN包被Nagle’s算法合并，会导致EPOLL的ET模式只出发一次，然而在应用层的SOCKET是读取返回0才代表链接关闭，而读取这次合并的数据包时是不返回0的，然后SOCKET以后都不会触发事件，所以导致应用层没有关闭SOCKET，从而产生大量的CLOSE_WAIT状态链接。
